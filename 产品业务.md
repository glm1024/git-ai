# Git AI 产品业务档案

## 业务目标

Git AI 是面向使用 AI 编码工具的研发团队的 Git 扩展，目标是在不改变日常提交流程的前提下，记录代码中哪些行由 AI 辅助生成，以及这些行对应的工具、模型、会话和提示上下文。产品重点不是事后猜测代码是否由 AI 生成，而是在代码产生和提交过程中采集来源事实，并让这些来源信息随 Git 历史一起流转。

## 核心角色

- 开发者：正常使用 AI 编码工具和 Git 提交代码，希望事后能理解某段代码的生成背景。
- 团队负责人和工程效能角色：关注 AI 代码采纳率、保留率、模型/工具效果和团队实践差异。
- 安全、合规和审计角色：需要知道 AI 参与了哪些代码变更，以及相关上下文是否可追溯、可控、可审查。

## 主要流程

1. AI 编码工具或编辑器在代码变更发生时触发检查点，记录文件内容、来源会话和模型等上下文。
2. 提交时将检查点汇总为行级作者来源记录，并作为 Git 附加元数据关联到提交。
3. 查询时通过统计、类 blame 视图或上下文问答，查看每行代码背后的 AI 工具、模型、会话和提示。
4. 当发生 rebase、squash、cherry-pick、stash、reset、amend 等 Git 历史改写或工作区迁移操作时，系统尝试让 AI 来源信息随代码内容迁移，而不是只停留在原始提交上。

## 关键业务规则

- 来源记录以实际采集事实为准，不把“检测 AI 代码”作为主要口径。
- 开源本地模式应尽量本地优先，避免把代码、提示或会话默认上传到外部服务。
- 提交历史不应被来源记录污染，来源信息应作为 Git 原生附加数据存在。
- 行级来源信息要能区分 AI、已知人工编辑和未知来源，不把未知来源简单等同于人工或 AI。
- 团队/企业场景可以在权限控制和脱敏基础上同步提示、会话和统计数据，用于审计、度量和协作。

## 重要边界

- 产品价值依赖 AI 工具、编辑器或 Git 流程中能采集到足够来源事实；未被采集的变更只能作为未知来源处理。
- Git 历史改写、服务端合并和文件重命名等场景会影响行级来源连续性，需要专门处理才能保持准确。
- 公开仓库和公开文档本身可能构成专利申请前公开披露；涉及专利布局时应尽快由专利代理人确认公开时间、申请地域和可主张范围。
